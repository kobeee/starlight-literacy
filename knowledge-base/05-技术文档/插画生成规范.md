# 插画生成规范

## 概述

本文用于约束当前项目的插画生成、恢复性补图和图片审阅工作流，重点解决两个问题：

1. 视觉稿误删后，如何按权威记录安全重建；
2. 多张原图直接进入上下文后，请求体过大、审阅失控的问题。

当前规范已进入生效状态，后续所有补图默认遵守本文。

## 权威输入优先级

生成或恢复插画时，输入来源按以下优先级解释：

1. 项目设计系统：`DESIGN.md`、`.impeccable.md`
2. 产品流程权威源：[[../01-产品设计/USER_FLOW|USER_FLOW]]
3. 最近的正式收口日志：尤其是 2026-04-22 ~ 2026-04-24 的 P01/P02/P03/P04/P05/P08/P09/P11 日志
4. 现存正式页面导出和已确认在用素材
5. 更早期的实验稿、旧导出、备份稿

如果旧图与新日志冲突，以**较新的正式日志**为准，不以旧导出为准。

## 恢复性补图规则

### 1. 恢复的是“最新正式状态”

遇到误删或缺图时，不直接追求“把旧图原样复刻”，而是：

- 先确认该资产最后一次正式收口的定义
- 再按最新正式状态重画

当前最典型案例：

- `P01` 幸存导出 `Qm8Jg` 仍是旧星空稿
- 但正式方向已切到“田园暖彩 + 暖阳绘本”
- 所以 `P01-v2` 必须按暖彩正式版恢复，不能按旧星空稿复刻

### 2. 先恢复结构型主资产，再做增强素材

补图优先级：

1. 页面主背景 / 长卷 / 大构图
2. 单屏视窗或关键状态预览
3. 局部增强插画或替换素材

不要先补局部装饰，再回头补主结构。

### 3. 页面批次独立推进

一次只处理一个页面批次。

推荐批次：

- Batch 1：`P01` 长卷与视窗
- Batch 2：必要时补 `P03` / `P09` 的局部增强插画
- Batch 3：必要时补组件或备用构图

## 请求体控制规范

### 硬规则

- 不把原始大图直接塞进生成请求
- 不把多张正式导出图一次性全部带入上下文
- 不在同一次请求里混入多个页面的原图

### 推荐做法

- 优先使用文字描述、页面职责、节点结构和颜色规范
- 如必须参考画面，只使用缩略图
- 缩略图优先控制在单边 `320px` 左右，足够看构图即可
- 需要多轮评审时，始终先看缩略图，再决定是否看原文件

## Prompt 组织规则

插画生成 prompt 应尽量结构化，至少包含：

- 用途：这个图给哪个页面/区域用
- 场景：花田、小屋、云朵、田园长卷、暖日庆祝等
- 主体：孩子、道路、节点空地、奖杯、字源角色等
- 风格：暖阳绘本、低饱和暖彩、手绘质感
- 约束：不要夜空蓝紫、不要霓虹 glow、不要高饱和彩虹平均分配

恢复性 prompt 还应明确：

- 保留哪些既有结构
- 本轮只修改什么
- 哪些旧方向不能回去

## 当前项目的固定视觉约束

### 允许

- 奶油米底
- 蜂蜜金主强调
- 暖棕主文字
- 草绿 / 樱粉 / 天蓝 / 橘黄等低饱和花田辅助色
- 花田、小屋、云朵、小动物、太阳暖光

### 禁止

- 紫蓝梦幻渐变
- 深蓝夜空底作为当前主视觉
- 霓虹 glow
- 平均分彩虹六色
- 为了“可爱”堆满烟花、粒子、贴纸

## 文件落地规则

- 生成后要把最终可用稿落回工作区，不只留在默认生成目录
- 不覆盖已有正式素材，除非明确确认替换
- 默认使用版本化命名，例如 `xxx-v2.png`、`xxx-clean.png`

## 示例

### 好的做法

- 先从日志提炼 P01 的节点、状态、构图和色板
- 只带一张 320px 缩略参考图做复核
- 单独生成 `P01` 长卷背景

### 不好的做法

- 把 `P01/P02/P03/P08/P09` 多张原图一次性塞进请求
- 一次要求同时补首页、认字页、结果页所有插画
- 明明正式方向已经改暖彩，还继续拿旧星空稿做主参考

## 注意事项

- 审阅图片前先压缩缩略图，避免大图内联进上下文
- 恢复工作默认保守，不直接覆盖旧文件
- 若文档与现存图冲突，先回溯日志，必要时暂停生成
- 如果用户明确说“请求里不要带原图”，必须只用文本 prompt 或本地缩略复核，不能把正式导出图、背景图或截图作为生成输入。
- P01 当前正例：`images/p01-pastoral-overlay-20260426.png` 使用纯文本 prompt 生成洋红 chroma-key 图，再本地抠色为透明叠层；该流程避免了原图请求体过大，也避免把旧动物形状继续补进正式稿。
