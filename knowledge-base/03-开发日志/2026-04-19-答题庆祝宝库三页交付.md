## 关键技术收获

### Pencil 渲染 bug 根因修正

**旧假设（错的）**：画布 x 坐标 > 1960 时，frame 的子节点不渲染。见 `Pencil-MCP设计技巧.md` 2026-04-15 节的"画布宽度渲染限制"。

**本轮反证**：
- ZXtU6（P09-付费引导）x=2940 → 渲染正常
- wEaOQ（P11-家长中心）x=3430 → 渲染正常
- 本轮新建的 P06/P07/P10 即便放到 x=0 也全部空白

**真因**：用 `I()` 从零创建包含多层子节点的**复杂 frame（即整页 390×844 结构）**会静默失败——文件里节点都存在、batch_get 能读出来，但 `get_screenshot` 和 `export_nodes` 都只能渲染父 frame 背景，所有子节点不可见。

**对照组**：
- ❌ `I("document", {type:"frame", width:390, height:844, layout:"vertical", padding:[...], ...})` 从零建整页 → 空白
- ✅ `C("JGbRx", "document", {x:4410, y:0})` 从现有 P05 Copy → 立即渲染
- ✅ 向已渲染的 frame 里用 `I()` 单节点（text / icon_font / 简单 rect）→ 渲染正常

**修正后的 Copy 策略工作流**：

```
新建复杂页面/卡片布局:
  1. 找结构最相似且当前渲染正常的节点 (JGbRx = P05 练习页是好模板)
  2. C(srcId, "document", {x, y}) 复制产出整棵子树
  3. U("newId/childId", {...}) 逐项改内容（文字/颜色/图标）
  4. 用不到的子树用 D() 删除

不再用 I() 从零建整页结构。
```

这个洞察和 4/16 记录的"flexbox 布局反复崩溃 → Copy 策略"本质是同一问题的两个面——**Pencil MCP 的渲染器对复杂嵌套结构的从零生成不够稳定，必须基于已工作的渲染上下文增量改动**。

### 本轮 memory 纠正

更新了 `feedback_pencil_canvas_limit.md`：
- 旧：x>1960 子节点不渲染，用 2×4 网格绕过
- 新：I() 从零建整页静默失败，必须 C() 复制模板再 U() 改

这条错误假设从 4/15 一路带到 4/19，中途浪费了不少调试时间追错方向（一度以为要移动 frame 位置才能让渲染恢复）。教训是：**memory 里的工具行为假设必须能被反证实验推翻**，不能因为"以前这么干过"就当成事实。
