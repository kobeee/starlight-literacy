# 技术选型决策

## 总体方案

| 层面 | 选型 | 理由 |
|------|------|------|
| 前端框架 | 微信小程序原生开发 | 性能最优，深度使用微信能力 |
| 渲染引擎 | Skyline | 同步光栅化、Worklet 动画、手势系统 |
| 组件框架 | glass-easel | Skyline 默认，建树性能优化 30-40% |
| 后端 | 微信云开发 | Serverless，免服务器、免备案 |
| 数据库 | 云数据库（JSON 文档型） | 与云开发深度集成 |
| 文件存储 | 云存储 | CDN 加速，直接上传/下载 |
| 支付 | 微信支付 | 小程序原生集成 |
| AI 内容生成 | Claude Opus 4.6 + Nano banana pro | 数据配置 + 插画生成 |
| 语音合成 | 待定（AI TTS） | 认读语音生成 |

## 为什么选原生而非跨端框架？

### 对比结论（2026 年 4 月数据）

| 指标 | 原生 | Uni-app 5.0 | Taro 3.8 |
|------|------|------------|----------|
| 冷启动 | 680ms | 760ms | 820ms |
| 内存占用 | 145MB | 162MB | 185MB |
| Skyline 支持 | 完整原生 | 需适配 | 需适配 |
| 动画性能 | 最优 | 接近 | 一般 |

**决策理由**：
1. 当前聚焦微信小程序，原生性能最优
2. 儿童识字 App 对动画性能要求极高（书写、拖拽、粒子特效）
3. Skyline 的 Worklet 动画和手势系统在原生开发下体验最佳
4. 独立开发者项目，原生开发的学习曲线可接受
5. 未来若做 iOS/Android，各端 UI 独立实现即可（AI 辅助重写成本低），不需要跨端框架

## Skyline 渲染引擎关键特性

- **同步光栅化**：避免快速滚动白屏，对学习地图卷轴至关重要
- **Worklet 动画**：渲染线程同步运行，书写页逐笔反馈无延迟
- **手势系统**：原生拖拽、缩放支持，适合拖拽匹配题型
- **共享元素动画**：页面转场平滑
- **WXSS 预编译**：样式解析速度提升 5x

### 环境要求
- 微信安卓/iOS 8.0.40+
- 基础库 3.0.2+
- 开发者工具 Stable 1.06.2307260+

## 云开发方案

### 云函数规划
- `user`：用户信息管理
- `learning`：学习进度同步
- `payment`：支付处理
- `share`：分享卡片生成
- `content`：内容数据管理

### 数据集合规划
- `users`：用户表
- `learning_progress`：学习进度
- `units`：单元数据
- `characters`：字库数据
- `orders`：订单记录

## 多端扩展策略

当前聚焦微信小程序，未来视用户反馈可扩展至 iOS / Android 原生 App。

**核心原则：不追求代码复用，共享资产和知识。**

| 跨端共享 | 各端独立 |
|---------|---------|
| 插画、音频等素材资源 | UI 界面和交互代码 |
| 字库、单元配置等 JSON 数据 | 平台原生动画和手势 |
| 系统设计文档（知识库） | 后端调用方式 |
| 业务规则描述（出题/评分逻辑） | 平台特有 API 调用 |

- 各端 UI 独立实现，AI 辅助重写成本远低于维护跨端框架的适配成本
- 代码分层（services/ + core/）是基本工程素养，保持代码整洁
- 素材和内容数据统一存放在项目根目录 `assets/` 和 `data/`，所有端共用
